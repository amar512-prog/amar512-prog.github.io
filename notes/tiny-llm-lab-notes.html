Tiny LLM lab notes from a local MLX training run

This page documents a compact GPT-style language model trained locally on Apple Silicon. It keeps the useful pieces together in one place: what the project already supports, how the model is built, which checkpoint is the safest one to reuse, and what to try next.

Current snapshot

The goal of this project is to train a very small decoder-only language model from scratch on Apple Silicon using MLX. The current working dataset is data/tiny_shakespeare.txt, the main tokenizer lives in src/tiny_llm/tokenizer.py, and the recommended checkpoint is checkpoints/shakespeare_bpe_seq/best.

Checkpoint Tokenizer Step Train loss Validation loss
shakespeare_bpe_seq/best BPE, vocab 512 20000 2.4669 3.0312

What the project already has

  • Character-level tokenization for the simplest end-to-end training path.
  • Byte-level BPE tokenization for a more realistic small-model setup.
  • GPT-style decoder blocks with causal self-attention.
  • Checkpoint save, resume, and best-checkpoint tracking.
  • Deterministic sequential evaluation for more stable checkpoint comparisons.
  • Sampling controls including top-k, top-p, repetition penalty, and repetition windows.

Model architecture in detail

The model is implemented as MiniGPT in src/tiny_llm/model.py. It is intentionally small enough to inspect end to end without losing the core ideas of a real Transformer.

Config field Value What it controls
block_size 128 Maximum number of previous tokens the model can use as context.
n_layer 4 Number of stacked Transformer blocks.
n_head 4 Number of attention heads inside each block.
n_embd 128 Width of the model representation for each token.
dropout 0.1 Regularization used in the embeddings, attention, and feed-forward layers.

End-to-end forward path

  1. The input arrives as token IDs with shape (batch, tokens).
  2. Each token ID becomes a learned token embedding of width 128.
  3. Each position gets its own learned position embedding, also width 128.
  4. Token and position embeddings are added together and passed through dropout.
  5. The representation flows through 4 Transformer blocks in sequence.
  6. A final LayerNorm is applied.
  7. A linear language-model head projects each position into vocab_size logits.

Transformer block layout

Each block uses a pre-norm residual structure:

x = x + attention(LayerNorm(x))
x = x + feed_forward(LayerNorm(x))

In practice, that means each block learns a refinement on top of the running representation instead of rebuilding the full state from scratch.

Attention details

  • A single linear layer creates query, key, and value projections together.
  • Those tensors are split into 4 attention heads.
  • Each head gets 32 channels because 128 / 4 = 32.
  • Attention scores are scaled by head_dim ** -0.5.
  • A lower-triangular causal mask blocks any lookahead into future tokens.
  • Softmax turns the masked scores into attention weights.
  • The head outputs are recombined, projected back to width 128, and passed through residual dropout.

Feed-forward path and generation loop

After attention, each block uses a two-layer MLP that expands the hidden width from 128 to 512, applies a GELU, then projects back down to 128. During generation, the model crops context to the most recent 128 tokens, uses the last-position logits, applies temperature and optional sampling controls, samples one token, appends it, and repeats.

Main milestones

1. Tiny smoke model

The earliest run existed mainly to prove that training, checkpoint saving, and generation all worked on a very small corpus. That checkpoint still lives in checkpoints/tiny.

2. Character model on Tiny Shakespeare

The first larger Shakespeare run used character tokens and reached a best validation loss of 1.6525 at step 5500 with a vocabulary size of 65. It learned line structure and speaker patterns well, but it still broke words apart because it predicted one character at a time.

3. BPE model on Tiny Shakespeare

The project then moved to byte-level BPE with a vocabulary size of 512. That made the model predict larger word pieces instead of isolated characters. The best checkpoint from this stage reached a validation loss of 3.0356 at step 17000.

4. Deterministic best-checkpoint tracking

The current recommended run uses deterministic sequential evaluation, which makes validation comparisons repeatable instead of noisy. That is why checkpoints/shakespeare_bpe_seq/best is the safest default checkpoint to reuse for generation examples or follow-up training.

Why the current checkpoint matters

One of the practical lessons from this project was that resumed training can quietly drift if the context length is not carried forward correctly. The current BPE sequential checkpoint reflects the later fixes: stable context handling, cleaner evaluation, and checkpoint comparisons that are easier to trust.

A smaller loss on the character model does not automatically make it the better checkpoint. Those losses are measured over different token spaces, so comparisons are most meaningful within the same tokenizer setup.

Recommended commands

Setup

python3 -m venv .venv
.venv/bin/python -m pip install -r requirements.txt
.venv/bin/python setup.py develop

Download Tiny Shakespeare

.venv/bin/python scripts/download_tiny_shakespeare.py

Run tests

.venv/bin/python -m unittest discover -s tests

In restricted sandboxed environments, MLX can still fail during Metal initialization. If that happens, rerun the verification directly on the host Mac before treating it as a code regression.

Train a fresh BPE run

.venv/bin/python -m tiny_llm.train \
  --data data/tiny_shakespeare.txt \
  --out-dir checkpoints/shakespeare_bpe \
  --tokenizer bpe \
  --vocab-size 512 \
  --max-steps 1500 \
  --eval-interval 300 \
  --eval-iters 20 \
  --batch-size 32 \
  --block-size 128 \
  --n-layer 4 \
  --n-head 4 \
  --n-embd 128 \
  --device gpu

Continue from the current best deterministic checkpoint

.venv/bin/python -m tiny_llm.train \
  --data data/tiny_shakespeare.txt \
  --init-from checkpoints/shakespeare_bpe_seq/best \
  --out-dir checkpoints/shakespeare_bpe_seq \
  --max-steps 2000 \
  --eval-interval 1000 \
  --eval-mode sequential \
  --batch-size 32 \
  --learning-rate 4e-5 \
  --lr-schedule cosine \
  --min-learning-rate 8e-6 \
  --warmup-steps 100 \
  --grad-clip 1.0 \
  --device gpu

Generate text from the recommended checkpoint

.venv/bin/python -m tiny_llm.generate \
  --checkpoint checkpoints/shakespeare_bpe_seq/best \
  --prompt "KING HENRY:" \
  --max-new-tokens 250 \
  --temperature 0.4 \
  --top-k 12 \
  --top-p 0.85 \
  --repetition-penalty 1.2 \
  --repetition-window 96 \
  --device gpu

Best next experiments

  • Train a slightly wider or deeper model while keeping the same evaluation discipline.
  • Compare several BPE vocabulary sizes instead of only 512.
  • Automate plots from metrics.csv so progress is easier to inspect.
  • Try a second dataset such as TinyStories for cleaner language patterns.
  • Keep deterministic validation as the default path for checkpoint decisions.

The reason this note matters is not that the model is huge. It is that the project is small enough to understand fully while still containing the real moving parts of a GPT-style system: tokenization, attention, residual blocks, checkpoints, validation, and sampling.
